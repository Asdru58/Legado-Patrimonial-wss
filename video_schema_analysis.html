Análisis Arquitectónico: Esquema de Video para Conferencias

Proyecto: Legado Patrimonial WSS  |  Fase 5.6 – 5.7  |  Marzo 2026

Contexto actual del proyecto

El esquema existente ya separa conferencias y multimedia en dos tablas con relación 1:1 (una fila en multimedia por conferencia). La tabla multimedia tiene: audio_url, video_url, pdf_url. El service layer en conferences.ts usa embedded selects de Supabase PostgREST para traer la multimedia en una sola query.

Requerimiento: Soportar un proveedor principal (YouTube) + fallbacks (R2/S3) para video, con capacidad de escalar a miles de conferencias.

Opción A — Esquema Plano (columnas directas en multimedia)

Añadir columnas: video_provider, youtube_id, video_fallback_url a la tabla multimedia existente.

CriterioEvaluación
Rendimiento✅ Excelente. Indexación directa sobre youtube_id. Sin JOINs extra.
Mantenimiento (Admin)✅ Formularios simples: inputs de texto estándar.
Queries Next.js✅ Zero-friction con el patrón actual de conferences.ts. Solo se agregan campos al SELECT.
Escalabilidad de providers⚠️ Limitada si se necesitan 3+ fallbacks con metadata distinta.
Migración✅ Una sola ALTER TABLE.

Veredicto A: Funcional y simple, pero rígida si los requisitos crecen.

Opción B — Esquema Relacional (tabla video_sources)

Nueva tabla video_sources(id, conferencia_id FK, provider, url, quality, is_primary, created_at).

CriterioEvaluación
Rendimiento✅ Bueno. JOIN adicional, costo mínimo con índice en conferencia_id.
Mantenimiento (Admin)⚠️ Necesita UI de sub-formulario dinámico (agregar/quitar sources).
Queries Next.js⚠️ Requiere nested select adicional y lógica de fallback en frontend.
Escalabilidad de providers✅ Excelente. N providers sin cambiar esquema.
Migración⚠️ Crear tabla + FK + índice + migrar data existente.

Veredicto B: Máxima flexibilidad, pero introduce complejidad que este proyecto no necesita hoy.

Opción C — Columna JSONB

Añadir video_sources JSONB en multimedia con estructura JSON anidada.

CriterioEvaluación
Rendimiento✅ Bueno. GIN index sobre JSONB si necesitas queries por provider.
Mantenimiento (Admin)⚠️ Necesita editor JSON estructurado o builder custom.
Queries Next.js⚠️ PostgREST no soporta deep JSONB filtering cómodamente. Requiere RPC.
Type-safety⚠️ Sin garantía de esquema a nivel DB. Mitigable con Zod.
Migración✅ Una sola ALTER TABLE.

Veredicto C: Elegante en teoría, pero rompe la armonía con el patrón PostgREST actual.

Recomendación definitiva

✅ OPCIÓN A (Esquema Plano) — con una variante ligeramente mejorada.

Justificación

  • El proyecto ya tiene multimedia con relación 1:1. Añadir columnas ahí es el camino de menor fricción.
  • El alcance real es acotado: un proveedor principal (YouTube) + uno o dos fallbacks.
  • El service layer actual funciona con embedded selects limpios. La Opción A preserva ese patrón exacto.
  • El futuro panel Admin se beneficia de formularios simples con inputs directos.

SQL de migración propuesto

ALTER TABLE multimedia
  ADD COLUMN video_provider TEXT DEFAULT 'youtube',
  ADD COLUMN video_provider_id TEXT,          -- ej: "dQw4w9WgXcQ"
  ADD COLUMN video_fallback_url TEXT,         -- ej: "https://r2.legado.../video.mp4"
  ADD COLUMN video_fallback_provider TEXT;    -- ej: "r2", "s3"

Función de resolución de video (utility)

function resolveVideoUrl(multimedia: Multimedia | null): string | null {
  if (!multimedia) return null;

  // 1. Intentar provider principal
  if (multimedia.video_provider === 'youtube' && multimedia.video_provider_id) {
    return `https://www.youtube.com/embed/${multimedia.video_provider_id}`;
  }

  // 2. Fallback directo
  if (multimedia.video_fallback_url) {
    return multimedia.video_fallback_url;
  }

  // 3. Legacy: video_url existente
  return multimedia.video_url;
}
